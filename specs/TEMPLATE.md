## Summary

One paragraph: what is this feature or architectural change, what problem does it solve, and why does it matter?

## Intent

### Goal

State the primary objective in one sentence.
Define the key abstractions, types, APIs, crate boundaries, or protocol surfaces this change introduces or modifies.

### Invariants

List the correctness, safety, compatibility, protocol, serialization, transcript, or performance invariants the implementation must preserve.
For prover/verifier changes, explicitly state the prover/verifier consistency requirements.

### Non-Goals

List what is explicitly out of scope.
Good non-goals prevent scope creep and clarify what reviewers should not expect from this PR.

## Evaluation

### Acceptance Criteria

Concrete, testable criteria.
Each item should be verifiable by a test, benchmark, assertion, compile check, or reviewable artifact.

- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Criterion 3

### Testing Strategy

Which existing tests must continue passing?
What new tests are needed?
Specify feature combinations, release/debug requirements, or deterministic fixtures when relevant.

### Performance

State performance expectations and the measurements that verify them.
For pure refactors, "no regression" is acceptable if the spec names the benchmark or metric.

## Design

### Architecture

Describe how this fits into the existing system.
Name the modules, crates, traits, data structures, and boundaries affected.
Include a diagram if the interaction is non-trivial.

### Alternatives Considered

What other approaches were evaluated?
Why was this design chosen over them?

## Documentation

List README, docs, examples, or developer-guide changes required.
If no documentation changes are needed, say why.

## Execution

Optional implementation guidance: suggested order of changes, modules to touch, migration notes, or risks to resolve first.

## References

Links to papers, related specs, issues, PRs, prior art, or local design notes.
